AI Bottleneck Diagnostic — Track C

This is the live, minimal version of the tool. The diagnostic sentence is produced by a human (per the Move 3 boundary decision below), and saved into a real, row-level-secured Postgres database. Below the tool, this page also documents the 2-user RLS isolation test required for Move 4.

Run a diagnosis

Enter the subject's name and the diagnostic sentence you (the human diagnostician) produced for them. This writes a real row into the live database, scoped to this demo account via row-level security — exactly the same mechanism tested below.

Schema

Tables: sessions, interview_transcripts, diagnoses, reactions. The critical diagnosis→outcome link is reactions.diagnosis_id → diagnoses.id. Row-level security is enabled on all four tables, scoped via sessions.user_id = auth.uid() and cascaded through the foreign keys.

Test 1 — User A inserts a session

User A authenticates and creates a session row. The user_id column is auto-populated from their auth token (default auth.uid()), not passed manually.

Test 2 — User B attempts to read all sessions (cross-read attempt)

User B, a completely separate authenticated account, requests all rows from the sessions table — the same table User A's data lives in.

HTTP/1.1 200 OK
[]

PASS — User B receives an empty array. User A's session is invisible to User B, enforced by Postgres row-level security, not application-level filtering.
